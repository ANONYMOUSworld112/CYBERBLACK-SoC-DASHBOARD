# Security

Threat model, authentication model, and hardening checklist for CYBERBLACK-SoC-DASHBOARD v2.0.

> This document covers what we ship, what we deliberately do not ship, and what you must do before exposing the dashboard outside the loopback interface.

---

## 1. Threat model

### What we defend against

| Threat | Defense |
|---|---|
| **Credential stuffing / brute force on `/login`** | bcrypt 12 rounds + per-IP rate limit (5 attempts / minute) + per-user lockout after 10 failed attempts in 15 min |
| **JWT theft** | 12-hour TTL, server-side session table for revocation, `Authorization: Bearer` only (no cookies), `Strict-Transport-Security` header in production |
| **2FA bypass** | `requireTotp` middleware on every resource route. The JWT itself does **not** assert TOTP â€” verification is re-checked against the DB on every request. |
| **XSS** | Helmet CSP, React 18 default-escaping, no `dangerouslySetInnerHTML` in the codebase, no inline scripts |
| **CSRF** | Bearer tokens (not cookies) â€” no CSRF surface |
| **Clickjacking** | `X-Frame-Options: DENY` (Helmet default) |
| **SQL injection** | All queries are prepared statements via the shim. No string interpolation of user input. |
| **Prototype pollution** | `express.json({ limit: '1mb' })` â€” no `qs` deep-merge exposure. Helmet's `crossOriginEmbedderPolicy` off only because we need `data:` URIs for QR codes. |
| **Open redirect** | CORS allow-list is a single origin from `CLIENT_URL`. No `redirect=` query param on any route. |
| **Information disclosure** | `/api/health` returns `{ok, ts}` only. No stack traces in 500 responses. Logs redact `Authorization` headers. |
| **Replay attacks on TOTP** | otplib enforces Â±1 window (30 s) by default. Reused codes within the window are rejected by the user-side `previousCode` check (server-side: 8-digit codes are out of scope; we ship 6). |
| **Privilege escalation** | Role check on every protected route. Default role is `analyst`. Admin-only routes are namespaced `/api/admin/*` (planned for v2.1; currently `admin` is a single user). |

### What we do **not** defend against

| Threat | Posture | Rationale |
|---|---|---|
| **Compromise of the host running the server** | Out of scope | If the attacker has shell, they have `data/soc.db` and the JWT secret. Mitigate with full-disk encryption, locked-down OS, and physical security. |
| **Insider with admin role** | Out of scope | v2.1 ships fine-grained RBAC; v2.0 ships a single admin. Use audit log. |
| **DoS / DDoS** | Out of scope | Deploy behind a reverse proxy with rate limiting + WAF. See [`DEPLOYMENT.md`](DEPLOYMENT.md). |
| **Side-channel (timing) on bcrypt compare** | Mitigated | bcrypt is constant-time on the hash itself; the path returns the same shape for "user not found" vs "wrong password" (both 401 `INVALID_CREDENTIALS`). |
| **Supply chain on npm deps** | Partial | We use pinned versions (`^`) and lockfile (`package-lock.json`). v2.1 adds `npm audit` to CI. |

### JWT properties

| Property | Value |
|---|---|
| Algorithm | HS256 |
| TTL | 12 h (`JWT_TTL` env, accepts any `ms` / `s` / `m` / `h` string) |
| Secret | 64 random bytes, base64url, generated by `setup.js` |
| Claims | `{ sub, role, jti, iat, exp }` |
| Storage (client) | `localStorage` under key `auth-storage` (Zustand persist) |
| Revocation | Server-side `sessions` table â€” `attachUser` checks `expires_at > now()` |

> The `totpVerified` claim is **not** in the JWT. It is re-checked from the DB on every request. This means a leaked JWT does not bypass 2FA, and revoking TOTP enrollment immediately blocks access.

### Password storage

- **Algorithm:** bcrypt
- **Cost:** 12 rounds (~250 ms on a modern server)
- **Salt:** per-user, generated by bcrypt
- **Comparison timing:** constant (bcrypt's design)
- **Storage:** `users.password_hash` (TEXT, 60 bytes)

### Rate limits

| Bucket | Limit | Window | Response |
|---|---|---|---|
| `/api/*` per IP | 100 req | 15 min sliding | 429 with `Retry-After` |
| `/api/auth/login` per IP | 5 attempts | 1 min | 429 |
| `/api/auth/verify-2fa` per IP | 10 attempts | 5 min | 429 |

Rate-limit storage is in-memory (per process). For multi-instance deployments, swap for a Redis-backed store (planned for v2.2).

## 3. Transport security

### Headers (Helmet defaults + our additions)

```js
helmet({
  contentSecurityPolicy: {
    defaultSrc: ["'self'"],
    scriptSrc:  ["'self'", 'https://fonts.googleapis.com'],
    styleSrc:   ["'self'", "'unsafe-inline'", 'https://fonts.googleapis.com'],
    fontSrc:    ["'self'", 'https://fonts.gstatic.com', 'data:'],
    imgSrc:     ["'self'", 'data:', 'https://*.basemaps.cartocdn.com'],
    connectSrc: ["'self'", 'ws:', 'wss:', 'https://*.basemaps.cartocdn.com'],
  },
  crossOriginEmbedderPolicy: false,  // QR codes use data: URIs
});
```

### CORS

```js
cors({
  origin: config.clientUrl,         // single origin from CLIENT_URL env
  credentials: true,
});
```

A misconfigured `CLIENT_URL` is the most common deployment mistake. See [`DEPLOYMENT.md`](DEPLOYMENT.md).

### TLS

- **Development:** loopback only, HTTP â€” fine.
- **Production:** terminate TLS at a reverse proxy (nginx, Caddy, ALB, Cloudflare). The Express server is HTTP behind it.
- **HSTS:** set at the reverse proxy (`Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`).

## 5. Secrets management

| Secret | Where | Rotation |
|---|---|---|
| `JWT_SECRET` | `.env` | `setup.js` generates a new one if `.env` is missing. To rotate manually: delete `.env`, re-run `setup.js`, restart server. **All existing sessions are invalidated.** |
| `ADMIN_PASSWORD` | `.env` | Set in `.env` before `setup.js` first runs. Change via `POST /api/auth/change-password` after first login. |
| `VT_API_KEY`, `SHODAN_API_KEY` | `.env` | Just update the file and restart. |
| `data/soc.db` (contains password hashes + TOTP secrets) | `data/` | File-system permissions. Production: `chmod 600 data/soc.db` or run the server under a dedicated OS user. |

> **Never commit `.env` or `data/`.** The `.gitignore` ships with both. Verify before pushing.

## 6. Hardening checklist (production)

- [ ] `JWT_SECRET` rotated from the dev default
- [ ] `ADMIN_PASSWORD` changed from `ChangeMe!2026`
- [ ] TLS terminated at a reverse proxy with HSTS
- [ ] `CLIENT_URL` set to the public origin
- [ ] `NODE_ENV=production`
- [ ] Server runs under a non-root, dedicated OS user
- [ ] `data/` is `chmod 700`, `data/soc.db` is `chmod 600`
- [ ] Backup of `data/soc.db` to off-host storage (daily)
- [ ] Reverse proxy rate-limits (independent of Express's)
- [ ] Firewall blocks port 4000 from anything but the proxy
- [ ] Log shipping configured (Winston JSON â†’ CloudWatch / Datadog / ELK)
- [ ] Monitoring on `/api/health` and `data/soc.db` file size
- [ ] 2FA enrollment enforced (this is on by default; verify no bypass exists)
- [ ] Audit log reviewed weekly

## 8. Compliance notes

CYBERBLACK-SoC-DASHBOARD v2.0 is a defensive tool. It does not include offensive capabilities (no exploit modules, no credential-stuffing tooling, no exfiltration helpers). The scanner tools (`nmap`, `vuln`) are intended to be run against **infrastructure you own or are authorized to test**. The OSINT tools query public WHOIS / DNS / Shodan and do not bypass authentication on third-party systems.

If you operate the dashboard in a regulated environment (SOC 2, ISO 27001, PCI-DSS, HIPAA, GDPR), the architecture supports it:

- **Audit trail:** every privileged action is logged.
- **Access control:** JWT + 2FA + role.
- **Data minimization:** the dashboard does not exfiltrate data. All lookups are user-initiated.
- **Right to erasure:** users are rows in `users`; deletion is a single `DELETE` plus token revocation.
- **Encryption at rest:** depends on the host's full-disk encryption (out of our scope).
- **Encryption in transit:** TLS at the reverse proxy.
